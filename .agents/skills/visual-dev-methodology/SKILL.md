# 视觉开发方法论(执行者版)

本 SKILL 提炼自一个真实项目:目标是在 Godot 中复刻一张 AI 概念图级别的"云海体素仙山"。
前十几轮在错误层级上迭代,进展为零;切换到本方法论后,约 20 轮达到目标图 78% 完成度。
本文写给**执行者**(干活的 agent)。评审裁决由独立的评审方完成(见"两级通过门")。

## 第一原则:先拆解,再开火

拿到目标图/参考风格,第一步永远是**观感构成拆解**:这张图的"高级感"由哪几层构成,各占多少权重。
例(本案的拆解,供参照格式):

| 层 | 权重 | 说明 |
|---|---|---|
| 渲染与光照(GI/大气/天空/影调) | ~40% | "好看的截图"的第一来源是 shader,不是贴图 |
| 体积云/雾等氛围系统 | ~20% | 有受光面和厚度的氛围,不是贴片 |
| 几何形体与分辨率 | ~25% | 形体读作什么(山/建筑/玩具)+ 体素/网格密度 |
| 植被与内容密度 | 包含于上 | 拆解时常被低估,需显式列出 |
| 贴图本身 | ~10% | 占比最小,却最容易被无限迭代 |

**火力必须打在权重最大的未完成层。** 在 10% 的层上迭代十轮,数学上不可能让画面逼近目标。
警告信号:如果你正在生成"候选 A vs 候选 B"这类同层平行方案超过 2 轮,几乎可以确定打错了层。

## 资产策略:关闭问题,而不是迭代问题

优先级顺序:
1. **CC0 现成资产**(ambientCG / Poly Haven 等照片扫描 PBR):质量碾压任何 AI 生成贴图,合法商用。选择限时 10 分钟,选错的代价为零(都是旋钮可调的)。
2. **AI 生图作为设定稿/贴图源**:适用于"品味问题"(造型、纹饰、比例)。生图产物先交评审,通过后才进入施工;生图当图纸或贴图,不当 3D 资产本身。
3. **程序化生成**:适用于大批量、参数化的内容(地形、植被撒布)。
4. **手工拼几何体**:最后手段。"用代码拼出有机/华丽造型"这条路基本必败,两次重试只会得到更多失败品。

## 轮次纪律(硬性)

- **一轮只动一个子系统。** 跨子系统改动必须有评审方的明确授权。
- **每轮报告必含两件套:**
  - **Uniform/参数 Diff**:本轮动过的所有参数,旧值 → 新值。漏报视为未申报漂移(严重违规)。
  - **Compare Grid**:目标图与本轮截图并排 + 双方 480px 缩略图。缺失视为本轮无效。
- **失败旋钮限额:** 每轮指定的可调旋钮,最多重试 2 次。用满未过门 → 停下,报告实际现象,等待评审。禁止自行扩大调参范围。
- **changed/unchanged 申报:** 报告中明确列出"改了什么 / 没碰什么"。
- **截图标准:** 固定机位预设;按截图等级取帧数: `FINAL=240` 帧(仅 hero 终验图与 compare_grid),`REVIEW=90` 帧(常规验收图: closeup/对比/帧条/环拍),`DEBUG=30` 帧(诊断图: ROI diff/overlay/红绿检测)。同一任务的截图必须在同一引擎会话内批量完成,禁止逐图重启;报告总截图耗时。

## 禁令(每一条都来自真实翻车)

1. **禁止参数作弊过门。** 把参数设到合法域之外(如坡度阈值设负数)让通过门"看起来通过"——会在更近的机位破相,且摧毁参数语义。同一数值在不同作用域可能合法(本案 cap_slope_min=-0.25 被锁定在峰顶帽檐带内时是合法设计),但必须申报语义并获评审认可。
2. **禁止未申报漂移。** 任何参数改动,无论多小、无论动机多好,必须出现在 Diff 里。本案 rock_tint 的一次未申报改动,导致后续两轮在错误基准上调光,全部白费。
3. **工程产出 ≠ 美术进展。** "功能正常/检测通过/管线跑通"只证明"没坏";只有 compare grid 的并排相似度才证明"够好"。自检报告里禁止用工程指标冒充观感判定。
4. **颜色/tint 类参数必须在最终光照下标定。** 无光或中性光环境下调好的颜色,在最终暖光/冷光下必然漂移。
5. **新网格生成路径上线前必跑"自检三件套"**(十分钟,换回的是数轮排查):
   - normal debug 图(法线方向可视化);
   - FRONT_FACING 红绿图(`cull_disabled + unshaded`,`ALBEDO = FRONT_FACING ? 绿 : 红`,可见面必须全绿);
   - 受光白材质图(纯白 StandardMaterial,验证光照真正到达)。
6. **审计要审到渲染结果,不是参数值。** "参数已设为 X"不等于"画面呈现 X"——中间隔着 modulate、vertex color、光照响应(roughness/specular 反射环境)、后期链路。画面与参数不符时,先出 unshaded debug 图隔离问题层,再修。

## 两级通过门与升级协议

每个阶段的通过门分两级:

- **工程门(自查自放行):** 可机器验证的项——检测图全绿、参数在合法域、Diff 完整、性能在预算内、无报错。
- **观感门(必须外部裁决):** 并排相似度、形体品味、色彩感受。执行者**无权**给观感门打 PASS。完成工作后停在交付点,提交 compare grid + Diff + 自检(自检可以写 CONCERNS,诚实的 CONCERNS 比虚假的 PASS 有价值得多),等待 D-director / director-review 裁决。

**为什么:** 本项目数据表明,执行者自检对观感缺陷的漏报率极高(云的硬切边、悬空树、检测覆盖缺口、造型失败等均为评审方发现)。这是结构性的——生产者验收自己的产出会继承同一套盲区——与执行者能力无关。

## 失败模式速查(症状 → 教训)

| 症状 | 教训 |
|---|---|
| 十几轮无进展、每轮都"有产出" | 在低权重层迭代;回到观感构成拆解 |
| 通过门全过但越来越不像目标 | 用工程门冒充了观感门;强制 compare grid |
| 调 A 参数画面几乎不变 | 该参数的作用通道已被上游压死(如乘法 tint 救不了饱和底色;改用去饱和+重染色) |
| 颜色反复二分仍不收敛 | 在错误光照下标定;或链路上有未知调制层,先 debug 图隔离 |
| 形体华丽但"身份"丢失(如体素感消失) | 实现层走捷径(直接出曲面);回到正确管线(先栅格化再出面) |
| 远处物体死色/不受光 | 跑自检三件套,大概率是背面/法线/层掩码问题 |
